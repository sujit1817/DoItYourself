🛒 Real World Walkthrough
User places Order #4521 on your e-commerce site — here's exactly what happens, step by step
🧑
Step 1 — User hits the API
POST /orders is called with { item: "MacBook", qty: 1 }
This triggers OrderService.createOrder(request)
💾
Step 2 — Order saved to DB
saveOrder(request) persists the order. Database gives back Order { id: 4521, status: PENDING }
📨
Step 3 — Event created & published
new OrderCreatedEvent(order) creates the event payload.
kafkaTemplate.send("order-events", "4521", event) publishes JSON to Kafka topic. Kafka stores it in partition determined by key "4521"
⚡
Step 4 — Kafka stores the message
Kafka logs: { "orderId": 4521, "item": "MacBook", "total": 1299, "userId": "u99" } to the order-events topic. OrderService returns immediately — no waiting!
📧
Step 5 — EmailService receives & acts
handleOrderCreated(event) fires automatically in EmailService. sendOrderConfirmation(event) emails the customer: "Your order #4521 is confirmed! 🎉"
Kafka marks the offset as consumed for group email-service
▸ Key Concepts at a Glance
📋
Topic
A named channel in Kafka. Like a radio station — producers broadcast, consumers tune in. Here: "order-events"
🔑
Message Key
Determines which partition the message lands in. Same key = same partition = ordered delivery for that order.
👥
Consumer Group
"email-service" group. Run 3 EmailService instances? Each handles different partitions — automatic load balancing.
🔄
Serializer
Converts Java objects ↔ bytes for network transport. Producer uses JsonSerializer, consumer deserializes back to the class.
🏭
ProducerFactory
Creates Kafka producer instances with your config. KafkaTemplate wraps this and gives you the clean .send() API.
⚡
Async / Decoupled
OrderService doesn't wait for email to send. They're completely independent — EmailService can be down and orders still work.
⚠️
What if Kafka goes down?
In this basic setup, if Kafka is unreachable when kafkaTemplate.send() is called, it throws an exception and the event is lost. Production systems add retries (acks=all, retries=3), an outbox pattern (save event to DB first, publish separately), and dead-letter topics for failed consumer messages.
